# PreToolUse Hook System — Requirements Document

## 1. Project Overview

### 1.1 Background

DevOps engineers interact with a wide variety of command-line tools daily: DOS/Windows CMD, PowerShell, Linux shell (Bash/Zsh), Terraform for infrastructure-as-code, Docker and Docker Compose for container orchestration, Kubernetes (kubectl) for cluster management, and AWS CLI for cloud operations. These commands range from harmless read-only inspection queries to deeply destructive operations that can delete production databases, terminate running instances, or wipe disk volumes.

AI coding assistants like GitHub Copilot and Claude Code are increasingly being used as pair programmers in terminal environments. These agents generate shell commands on behalf of the user based on natural-language prompts. While this accelerates workflows, it introduces a significant risk surface: an AI agent may confidently suggest a command that appears correct but carries unintended destructive consequences (e.g., `rm -rf /var/lib/docker` instead of `rm -rf /var/lib/docker/tmp`, or `kubectl delete pods --all -A` when the user intended `kubectl get pods -A`).

### 1.2 Problem Statement

There is currently no standardized safety layer between AI-generated shell commands and their execution. When an AI agent suggests a command, the user must manually review every single sub-command for potential side effects before executing — even trivially safe commands like `docker ps` or `terraform plan`. This creates cognitive overhead that defeats the productivity benefits of using AI assistants in operational contexts.

### 1.3 Solution

The PreToolUse Hook System intercepts AI-generated shell commands before execution, classifies each command (or sub-command within a chain) as either read-only or modifying, and applies the following policy:

- **Read-only commands** are auto-approved for immediate execution without user interaction.
- **Modifying commands** trigger a manual approval prompt where the user must explicitly confirm before execution.
- When classification is uncertain (e.g., complex one-liners with ambiguous intent), the system defaults to requiring manual approval, following the principle of least privilege and fail-safe design.

### 1.4 Scope

This project targets two AI coding environments:

1. **GitHub Copilot** — via VS Code extension hook mechanism
2. **Claude Code** — via pretooluse hook configuration

The system supports two deployment models: local-first (user-specific settings that only affect the current machine) and repository-level (shared `.pretooluse` or `.github/hooks/` configuration that applies to all contributors). Both modes are supported simultaneously.

## 2. Functional Requirements

### FR-1: Command Interception

The hook system shall intercept every shell command generated by an AI agent before it is submitted for execution in the terminal. This includes:

- Single-line commands (e.g., `ls -la`)
- Multi-line commands and here-documents
- Chained commands using operators (`&&`, `||`, `;`, `|`)
- Commands wrapped in code blocks (markdown formatting)
- Compound commands with subshells, command substitution `$()`, backticks, process substitution `<()`
- Mixed-shell commands where DOS/PowerShell and Linux commands appear in the same block

**Intercept latency**: The interception mechanism itself shall add no more than 50ms of overhead before the classification engine begins analysis.

### FR-2: Command Classification Engine

The system shall implement a hybrid command classification approach with three layers:

#### FR-2.1 Pattern-Based Matching (Layer 1 — Fast Path)

A regex-based pattern matcher shall classify ~80% of commands using a prebuilt database of known commands stored in `test-cases.xml`. This layer uses exact command-name matching, subcommand detection, and flag analysis to determine if a command is read-only or modifying. This layer operates within < 50ms per command.

#### FR-2.2 Rule Engine (Layer 2 — Complex Commands)

A rule-based engine shall handle complex commands that cannot be classified by simple pattern matching:

- **Chained commands**: When multiple commands are joined with `&&`, `||`, or `;`, each sub-command is parsed and classified individually. If any sub-command is modifying, the entire chain requires manual approval.
- **Piped commands**: A pipe (`|`) between two read-only commands remains read-only (e.g., `ps aux | grep nginx`). However, if any stage in a pipe chain is modifying, the entire chain requires approval.
- **Command substitution**: Commands embedded in `$()` or backticks are recursively classified. A read-only wrapper with a modifying inner command (e.g., `echo "deleted: $(rm -rf /tmp/old)")`) is flagged as modifying.
- **Conditional execution**: `if`, `case`, and ternary constructs are analyzed for worst-case path — if any branch contains a modifying command, the entire block requires approval.

This rule engine layer operates within < 300ms per command chain. For PowerShell commands, the system may invoke the built-in PowerShell parser (`System.Management.Automation.Language.Parser`) to tokenize complex scripts accurately.

#### FR-2.3 Fallback (Layer 3 — Unknown Commands)

Commands that cannot be classified by either Layer 1 or Layer 2 shall default to requiring manual approval with an informational message: "This command could not be automatically classified. Please review before executing." This ensures a fail-safe behavior where uncertainty always defaults to safety.

### FR-3: Risk Level Classification

Modifying commands shall be further classified into risk levels based on the potential impact of their execution, as defined in `test-cases.xml`:

- **risk="low"** — Changes that are easily reversible or localized (e.g., creating files, copying data, setting environment variables for a session)
- **risk="medium"** — Changes that affect system configuration, service state, or persistent storage but are generally recoverable (e.g., starting/stopping services, modifying firewall rules, scaling deployments)
- **risk="high"** — Destructive operations that cause irreversible data loss, permanent resource deletion, security-compromising changes, or significant operational disruption (e.g., `rm -rf`, `terraform destroy`, `kubectl delete --force`, `aws rds delete-db-instance`)

The UI presentation of the approval prompt shall reflect risk level: high-risk commands display a prominent warning with explicit confirmation requirements.

### FR-4: Audit Logging

The system shall maintain a complete audit trail for compliance and debugging purposes:

#### FR-4.1 Command Log File

Every intercepted command shall be recorded in a JSON-formatted log file located at `~/.pretooluse/audit.log` (local) or `.pretooluse/audit.log` (repository-level). Each log entry shall include:

```json
{
  "timestamp": "2026-05-12T23:45:12.345Z",
  "agent": "claude-code",
  "command": "terraform plan -var-file=prod.tfvars",
  "classification": "read-only",
  "risk_level": null,
  "action": "auto_approved",
  "layer_used": "pattern_matching",
  "latency_ms": 12,
  "sessionId": "abc-123-def"
}
```

For commands requiring manual approval:

```json
{
  "timestamp": "2026-05-12T23:46:08.122Z",
  "agent": "github-copilot",
  "command": "kubectl delete pods --all -A --force",
  "classification": "modifying",
  "risk_level": "high",
  "action": "user_confirmed",
  "layer_used": "pattern_matching",
  "latency_ms": 8,
  "sessionId": "abc-123-def"
}
```

Possible values for the `action` field: `auto_approved`, `user_confirmed`, `user_rejected`, `fallback_to_manual`.

#### FR-4.2 Log Rotation

The audit log file shall be rotated when it exceeds 50MB, retaining the last 5 rotated files with gzip compression to prevent unlimited disk growth.

### FR-5: User Interface for Approval Prompts

When a modifying command is detected, the system shall present an approval prompt that displays:

1. The full command text
2. Its classification (modifying) and risk level (low/medium/high)
3. Which sub-component(s) within a chain triggered the flag (for complex commands)
4. A clear accept/reject action with keyboard shortcut support

High-risk commands may require an additional confirmation step (e.g., typing "CONFIRM" or pressing Enter twice) to prevent accidental execution of destructive operations.

## 3. Non-Functional Requirements

### NFR-1: Performance

| Metric | Target | Rationale |
|--------|--------|-----------|
| Total hook latency (simple command) | < 100ms | Pattern matching + minimal overhead |
| Total hook latency (complex chain) | < 500ms | Includes PowerShell parser, rule engine recursion |
| Commands per second (sustained) | > 20/sec | Must not bottleneck during rapid batch operations |
| Memory footprint | < 50MB | Hook runs in the same process space as the IDE/editor |
| Startup time (initial load) | < 2 seconds | Loading XML database, compiling regex patterns |

The system shall use lazy loading and caching strategies: the `test-cases.xml` database is loaded once at startup and cached in memory. Regex patterns are compiled once and reused. The PowerShell parser is only invoked when a command is detected as PowerShell syntax (not on every invocation).

### NFR-2: Accuracy

- **Read-only commands**: False positive rate (flagging a read-only command as modifying) shall be < 1%. This prevents user fatigue from unnecessary approval prompts during routine operations.
- **Modifying commands**: False negative rate (missing a modifying command) shall be 0% in the Layer 1 pattern matcher — any unrecognized command falls through to Layer 3 fallback, which always requires manual approval. The safety guarantee is: no modifying command is ever silently auto-approved.

### NFR-3: Extensibility

The system shall support adding new command domains (e.g., Azure CLI, GCP CLI, Ansible) without code changes — only by extending the `test-cases.xml` database and optionally adding rules to the rule engine configuration file. New platforms can be onboarded by contributing a single XML category group following the existing schema.

### NFR-4: Portability

The hook system shall function on all three major operating systems: Windows (PowerShell/CMD), macOS (Bash/Zsh), and Linux (Bash/Zsh). Platform-specific command detection logic accounts for path separator differences, quoting conventions, and shell operator variations across environments.

### NFR-5: Security

The audit log file shall not contain secrets, credentials, or sensitive data extracted from commands. If a command contains patterns matching common credential formats (e.g., `--password`, `-k <key>`, base64-encoded blocks in payloads), those segments are masked with `[REDACTED]` before logging.

## 4. Command Classification Rules by Domain

### 4.1 DOS / Windows CMD

**Read-only pattern**: Commands that query system state, list files, display content, inspect network configuration, or search text without writing to disk or changing configuration. Key examples: `dir`, `type`, `tree`, `findstr`, `where`, `ipconfig`, `ping`, `tracert`, `netstat`, `systeminfo`, `whoami`, `tasklist`, `sc query`.

**Modifying pattern**: Commands that create, delete, move files; alter system configuration; manage services; modify registry entries; change user accounts; or affect network security. Key examples: `del`, `rmdir`, `copy`, `xcopy`, `robocopy`, `taskkill`, `shutdown`, `net start/stop`, `reg add/delete`, `sc config/delete`, `netsh advfirewall`.

### 4.2 PowerShell

**Read-only pattern**: Verbs following the PowerShell naming convention — `Get-*`, `Test-*`, `Select-*`, `Where-*`, `Sort-*`, `Measure-*`, `Compare-*`, `Export-*` (read-mode), `Write-Host`. These cmdlets read state without side effects.

**Modifying pattern**: Verb-based heuristics:
- **High risk**: `Remove-*`, `Stop-*`, `Restart-*`, `Disable-*`, `Destroy-*`, `Clear-*`
- **Medium risk**: `New-*`, `Set-*`, `Add-*`, `Start-*`, `Copy-*`, `Move-*`, `Rename-*`, `Enable-*`
- **Low risk**: `Export-Csv`, `Out-File` (writes to local disk only)

**Unknown cmdlet fallback**: A PowerShell cmdlet not in the known database is classified by its verb. If the verb matches a modifying pattern above, it requires approval. Otherwise, it falls through to Layer 3 (manual).

### 4.3 Linux / Bash / Zsh

**Read-only pattern**: Commands that inspect system state: `ls`, `cat`, `head`, `tail`, `grep`, `awk` (without output redirection), `sort`, `uniq`, `wc`, `find` (without `-delete`, `-exec`), `ps`, `top`/`htop` (non-interactive mode), `df`, `du`, `free`, `uname`, `ss`, `ip addr`, `journalctl` (query mode), `curl -sI` (HEAD requests).

**Modifying pattern**: Commands that write to disk, change permissions, alter system configuration: `rm`, `mv`, `cp` (to protected paths), `chmod`, `chown`, `sed -i`, `systemctl start/stop/restart`, `useradd/userdel`, `apt install/remove`, `yum/dnf`, `iptables -A/-D`, `mount/umount`, shell redirection operators (`>`, `>>`) to system files.

**Redirection detection**: A read-only command followed by a redirection operator (e.g., `grep foo log.txt > output.csv`) is classified as modifying (low risk) because it writes to disk. Redirections to `/dev/null` or `/tmp/` are treated as low-risk temporary writes.

### 4.4 Terraform

**Read-only pattern**: Commands that inspect state or plans without altering infrastructure: `terraform plan`, `terraform state list`, `terraform state show`, `terraform output`, `terraform console`, `terraform fmt -check`, `terraform validate`, `terraform providers`, `terraform graph`.

**Modifying pattern**:
- **High risk**: `terraform apply` (creates/changes/destroys infrastructure), `terraform destroy` (removes all managed resources), `terraform state rm/mv` (alters tracking without warning), `terraform force-unlock` (can corrupt state)
- **Medium risk**: `terraform import` (adds external resources to tracking), `terraform workspace new/select`
- **Low risk**: `terraform fmt` (reformats files on disk only)

### 4.5 Docker & Docker Compose

**Read-only pattern**: Inspection and monitoring: `docker ps`, `docker inspect`, `docker logs`, `docker stats`, `docker images`, `docker history`, `docker diff`, `docker events`, `docker info`, `docker top`, `docker volume ls/inspect`, `docker network ls/inspect`, `docker compose config/ps/logs`.

**Security note**: CVE-2025-62725 demonstrated that `docker compose ps` can be exploited via path traversal to access host files when using maliciously crafted compose files. Commands that reference untrusted compose file paths should trigger a warning even if the subcommand itself is classified as read-only.

**Modifying pattern**:
- **High risk**: `docker rm -f` (force-remove with potential data loss), `docker stop/kill`, `docker volume rm` (destroys persistent data), `docker system prune -a --volumes` (mass cleanup)
- **Medium risk**: `docker run/start/restart`, `docker exec` (remote execution in container), `docker build`, `docker push`, `docker rmi`, `docker compose up/down`

### 4.6 Kubernetes (kubectl)

**Read-only pattern**: Inspection and monitoring: `kubectl get`, `kubectl describe`, `kubectl logs`, `kubectl top`, `kubectl cluster-info`, `kubectl api-resources`, `kubectl explain`, `kubectl rollout status`, `kubectl wait`, `kubectl auth can-i` (policy dry-run), `kubectl config view/current-context`.

**Modifying pattern**:
- **High risk**: `kubectl delete --force`, `kubectl edit` (direct state mutation), `kubectl drain` (node eviction)
- **Medium risk**: `kubectl apply`, `kubectl create`, `kubectl patch`, `kubectl scale`, `kubectl expose`, `kubectl exec` (remote execution in pod), `kubectl rollout undo/pause/resume`, `kubectl cordon/uncordon`
- **Low risk**: `kubectl config use-context` (local kubeconfig change)

### 4.7 AWS CLI

**Read-only pattern**: Commands that describe, list, or get resource metadata without changing anything: `aws s3 ls`, `aws s3api head-object`, `aws ec2 describe-*`, `aws lambda list-functions/get-function`, `aws cloudformation describe-stacks`, `aws iam list-*`, `aws rds describe-*`, `aws cloudwatch get-metric-data/describe-alarms`, `aws eks describe-cluster/list-clusters`.

**Modifying pattern**:
- **High risk**: `aws ec2 terminate-instances`, `aws s3 rm --recursive`, `aws cloudformation delete-stack`, `aws rds delete-db-instance --skip-final-snapshot`, `aws iam attach-user-policy` (security impact)
- **Medium risk**: `aws s3 cp/sync` (writes data), `aws ec2 start/stop-instances`, `aws lambda invoke`, `aws cloudformation create-stack/update-stack`, `aws lambda update-function-code`, `aws rds modify-db-instance`

## 5. Edge Cases and Complex Command Handling

### EC-1: Mixed Shell Commands

Commands that combine syntax from multiple shells (e.g., a batch script calling PowerShell via `powershell -Command "..."` within CMD) are decomposed by detecting the subshell invocation boundary. Each segment is classified independently using its native shell's rules. The overall classification is the union of all segments — if any segment is modifying, the entire command requires approval.

### EC-2: Obfuscated Commands

Commands that use base64 encoding (`base64 -d <<< "..." | bash`), character manipulation to hide intent, or variable indirection are automatically flagged as high-risk and require manual approval with a security warning banner. The system does not attempt to decode obfuscated payloads — it treats them as inherently untrustworthy by definition.

### EC-3: Idempotent Operations

Some commands are modifying but idempotent (e.g., `mkdir -p`, `docker pull`, `terraform plan`). These are classified as modifying (low risk) because they technically alter state, even if running them multiple times produces no additional change. The user retains full visibility and can auto-approve specific patterns via a personal allowlist configuration.

### EC-4: Dry-Run Flags

The system shall recognize dry-run flags that convert modifying commands to read-only operations:
- `terraform plan` (vs `terraform apply`) is already classified separately
- `rsync --dry-run` — modifies nothing, classified as read-only
- `apt-get install --simulate` / `-s` flag — read-only simulation of package installation
- `kubectl apply --dry-run=client|server` — validates without applying changes

When a recognized modifying command is accompanied by a recognized dry-run flag for that specific tool, the effective classification is downgraded to read-only.

### EC-5: User Allowlists and Denylists

A user-specific configuration file (`~/.pretooluse/config.json`) shall allow users to override default classifications:

```json
{
  "allowlist": [
    "docker ps *",
    "kubectl get *",
    "terraform plan *"
  ],
  "denylist": [
    "echo * > /etc/*"
  ]
}
```

Denylist entries always take precedence over allowlist. Denylisted commands always require manual approval regardless of their default classification. This allows teams to enforce stricter policies for specific operations in sensitive environments.

## 6. Deployment Model Requirements

### DM-1: Local Deployment (Default)

The hook installs into the user's local AI agent configuration:

- **GitHub Copilot**: Configured via VS Code user settings (`settings.json`) or `.vscode/` workspace hooks
- **Claude Code**: Configured via `~/.claude/settings.json` pretooluse hook registration

Local deployment is the default installation path and affects only the current user on the current machine. It does not require any repository changes, pull requests, or team coordination.

### DM-2: Repository-Level Deployment

A `.pretooluse/` directory at the repository root provides shared configuration:

```
.pretooluse/
  config.json          # Shared allowlist/denylist rules for all contributors
  commands.custom.xml  # Team-specific command overrides and additions
  README.md            # Documentation of team safety policies
```

When a contributor clones a repository with `.pretooluse/` configuration, the hook system prompts them to opt into the shared policy on first use. This allows teams to enforce consistent safety standards across all developers without requiring individual setup.

### DM-3: Configuration Merge Order

When both local and repository-level configurations exist, they are merged with this priority (highest to lowest):

1. User-local denylist
2. Repository denylist
3. Repository allowlist
4. User-local allowlist
5. Default built-in classification rules from `test-cases.xml`

## 7. Data Model: Command Classification Database

The system uses an XML-based database (`test-cases.xml`) as its authoritative source of truth for known commands. This format was chosen because:

- **CDATA sections** preserve multi-line command syntax, pipes, redirections, and special characters without escaping
- **Hierarchical grouping** by platform (DOS_CMD, PowerShell, Linux, Terraform, Docker, Kubernetes, AWS_CLI) enables fast lookup
- **Extensibility** allows new platforms to be added as new `<category-group>` elements
- **Human readability** facilitates team review and auditing of classification rules

Each command entry specifies:
- `name` — the canonical command name or pattern
- `description` — human-readable explanation of what it does
- `example` (within CDATA) — real-world usage example for training and debugging
- `category` attribute — `"read-only"` or `"modifying"`
- `risk` attribute (on modifying commands) — `"low"`, `"medium"`, or `"high"`

The XML database is loaded into memory at startup. Pattern-matching regex are compiled from the database entries once, and a hash map of command names to their classification is built for O(1) lookup performance.

## 8. Testing Strategy

### 8.1 Unit Tests

Each layer of the classification engine shall be unit-tested independently:

- **Pattern matching**: Verify that each entry in `test-cases.xml` produces the expected classification when passed through Layer 1
- **Rule engine**: Test all chaining operators (`&&`, `||`, `;`, `|`), command substitution, and conditional constructs against known outcomes
- **Risk classification**: Confirm risk level assignments match database definitions

### 8.2 Integration Tests

End-to-end tests shall simulate real AI agent workflows:

1. Feed a curated list of 50+ read-only commands and verify auto-approval without user interaction
2. Feed a curated list of 50+ modifying commands at each risk level and verify correct prompt behavior
3. Feed complex chained commands (mixed shells, pipes, subshells) and verify correct classification propagation
4. Verify audit log entries are written correctly for every scenario

### 8.3 Adversarial Tests

The system shall be tested against intentionally adversarial inputs:

- Commands designed to look read-only but hide modifying behavior (e.g., `ls; rm -rf /` hidden in a wrapped line)
- Unicode homoglyph attacks where visually similar characters substitute for command names
- Extremely long commands (>10KB) to test parser stability and timeout handling
- Nested command substitution ($($()) with multiple levels of depth

### 8.4 Performance Benchmarks

Benchmark tests shall verify that the hook meets its latency targets under load:

- Classify 1,000 simple commands in < 100 seconds (< 100ms avg)
- Classify 100 complex chains with PowerShell parsing in < 50 seconds (< 500ms avg)
- Measure memory usage after sustained operation to detect leaks

## 9. Implementation Phases

### Phase 1: Foundation (Weeks 1-2)
- Parse and load `test-cases.xml` into an in-memory classification database
- Implement Layer 1 pattern matcher with O(1) command name lookup
- Build basic interception hook for Claude Code
- Write unit tests covering all XML entries

### Phase 2: Rule Engine (Weeks 3-4)
- Implement Layer 2 rule engine for chained, piped, and substituted commands
- Integrate PowerShell language parser for PS-specific tokenization
- Implement dry-run flag detection (EC-4)
- Build audit logging infrastructure

### Phase 3: UI & GitHub Copilot Integration (Weeks 5-6)
- Design approval prompt UI with risk-level-aware presentation
- Implement GitHub Copilot VS Code hook integration
- Add user allowlist/denylist configuration support
- Write integration and adversarial test suites

### Phase 4: Repository Deployment & Hardening (Weeks 7-8)
- Implement `.pretooluse/` repository-level deployment model
- Add log rotation, credential masking in audit logs
- Performance benchmarking and optimization
- Documentation, team onboarding flow for shared configs

## 10. Success Criteria

The PreToolUse Hook System is considered production-ready when:

1. All commands in `test-cases.xml` are correctly classified by Layer 1 pattern matching (verified via automated test)
2. Zero modifying commands are auto-approved without user confirmation (safety invariant — verified by adversarial tests)
3. Average hook latency for simple commands is < 100ms and complex chains < 500ms
4. Both Claude Code and GitHub Copilot integrations pass integration tests on all three supported operating systems (Windows, macOS, Linux)
5. Audit log captures 100% of intercepted commands with correct metadata
6. A team can deploy a shared `.pretooluse/` configuration that new contributors adopt automatically upon first use
